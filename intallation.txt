Part 5: Validate Environment Variables with Zod:
Currently, the backend reads environment variables directly through process.env. A missing or invalid value could cause an error later during application execution.
We’ll create one validated configuration object and use it throughout the backend. Zod validates input against a defined schema, while dotenv loads values from .env into process.env.
1: Install Zod
npm install zod

Part 6: Structured Logging with Pino:
1. Application logger
2. HTTP request logging
3. Unique request IDs
4. Sensitive-header redaction
5. Proper database and server logs
6. Request IDs in error responses
pino-http attaches a child logger to each request as request.log, supports custom request-ID generation, and automatically logs request completion time. Its default numeric request ID is not ideal when an application eventually runs across multiple instances, so we’ll generate UUIDs instead.
Step 1: Install Pino packages
npm install pino pino-http
npm install --save-dev pino-pretty
We’ll use:
pino         → Application logger
pino-http    → Express request/response logger
pino-pretty  → Readable development logs
In production, Pino will output structured JSON. In development, pino-pretty will format logs for easier reading. Pino supports transports such as pino-pretty, and transports run separately from the main application thread.

Part 12: Refresh Session Model:
A JWT refresh token should not remain valid only because its signature is valid. We also need a database record so the token can be:

Revoked during logout
Rotated during token refresh
Invalidated after password changes
Invalidated when an account is blocked
Tracked across multiple devices

We will store only a SHA-256 hash of the refresh token—not the raw token.
Complete security flow:
User logs in
Refresh JWT generated
Raw token sent securely to client
SHA-256 token hash stored in MongoDB
Client later sends refresh token
JWT signature verified
Received token hashed
Database session located
Check revoked status
Check expiration
Accept, rotate, or reject

Part 16: Logout API:
Logout will:
Read refresh-token cookie
→ Verify it when possible
→ Find the matching refresh session
→ Revoke that session
→ Clear access and refresh cookies
→ Return success
